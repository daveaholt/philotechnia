## Mission

Make Philotechnia usable for a person — wrap the M2a/M2b engine in a Qt 6 alpha UI that lets a user create, open, edit, and close projects, browse records in a `QTableView`, edit individual records in a detail view, build a schema through a dedicated editor, and undo/redo through a `QListView`-rendered history bound to `CommandStack::history()`. Every UI interaction goes through a `Command` — nothing mutates state directly. The `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` UI-thread bridge from §8.5 becomes live in M3: the M2b parallel-open coordinator now has a UI thread to hand off to, and the record table is populated progressively from worker chunks without blocking the event loop. No source control UI ships at M3 — that's M4's scope. Rule-6 exhaustiveness constituent (2) — renderer coverage for every `ErrorState` — lands in the M3–M4 window as the source control UI takes shape, with at least the recovery-UI scaffolding present by the end of M3.

**Predecessor:** M2b — query, indexes, parallel open, open cache. **Successor:** M4 — source control UI (pending changes, commit panel with `ValidationIssue` surfacing, push/pull, branches, history, conflict resolution).

## Foundation

### Project in one paragraph

Philotechnia is a cross-platform C++ desktop application for structured record management. Projects are version-controlled directories of plain JSON files (one file per record, hash-prefix directories). The application includes a first-party source control UI built on an abstract `SourceControlProvider` interface — Git via libgit2 ships in v1; SVN and Perforce are later milestones. The UI is Qt. JSON is yyjson. All undo/redo and dirty-state tracking flows through a custom `CommandStack`, not Qt's built-in undo framework.

### Non-negotiable architectural rules (from CLAUDE.md)

Rules active in M3:

1. **The UI layer must never depend on a concrete `SourceControlProvider` implementation.** The M3 UI surfaces none of the source control panel — that's M4 — but the title-bar provider-name indicator (decision #17) is visible from M3. It takes the provider's display name through an abstract accessor only, never by `dynamic_cast` to `GitProvider` or by including a provider-implementation header.
2. **`CommandStack` owns all undo/redo logic. Qt does not.** `QUndoStack` is not used anywhere. The undo history `QListView` is a pure view over `CommandStack::history()`; a `QAbstractListModel` subclass adapts the vector of descriptions to Qt's model protocol. Entries at or before `CommandStack::commit_boundary()` are rendered visually distinguished (italic / dimmed) per §6.8.
3. **Qt must be dynamically linked.** The M0a-era assertion in `CMakeLists.txt` stays green. Adding Qt Widgets and any additional Qt modules (e.g., `Qt6::Gui`, `Qt6::Widgets`) uses the same `find_package(Qt6 COMPONENTS … REQUIRED)` + shared-linkage check pattern. Static Qt remains a configure-time `FATAL_ERROR`.
6. **The application must be fully self-sufficient for all source control states it creates.** Constituent (2) of the rule-6 CI check — renderer coverage for every `ErrorState` — lands in the M3–M4 window. M3 is the earlier of the two milestones it can land in; at minimum M3 ships the recovery-UI scaffolding and the test that walks `ErrorState` against registered renderers, exempting values in `kUnshippedErrorStates`. Full flow wiring can close out in M4 with the source control UI, but the coverage test should be green (or green with known-failing renderers stubbed) by the end of M3.
9. **Commands mutate only in-memory state; disk I/O happens exclusively in the commit pipeline.** Every UI interaction that mutates project state constructs a `Command`, pushes it through `CommandStack::push()`, and lets the commit pipeline (M2a, not invoked in M3 except programmatically) handle persistence. The M3 inline table editor, the detail-view form fields, and the schema editor all produce commands — no UI code path calls `atomic_write` directly.

Rules 4, 5, 7, 8 are background: rule 4 (atomic rename) is already enforced in the commit pipeline; rule 5 (no ASM) is passively observed; rule 7 (provider-aware UI guidance) lands properly with the source control UI in M4 — M3 only touches the title-bar provider indicator and the recovery-UI scaffolding; rule 8 (commit-time SchemaValidator) already runs, though its user-facing surfacing is M4's job.

### Storage format (§7.1 summary)

Project = directory with `manifest.json`, `schema/...`, `records/{entity-type-id}/{xx}/{record-uuid}.json` (hash-prefix dirs). The M3 UI reads through M2a's `RecordManager` / M2b's `RecordIndex` — never `std::filesystem` directly. File paths are an implementation detail of the core; UI code sees `RecordId` and `EntityTypeId`.

### Error type (§8.7 summary)

`Error::Category` — `Io`, `NotFound`, `Parse`, `Validation`, `SourceControl`, `Authentication`, `Network`, `Conflict`, `FormatVersion`, `Cancelled`, `Internal`. The UI layer consumes `std::expected<T, Error>` and renders the failure as an inline error state per the category boundaries (§8.7):

- `Io`, `Parse`, `Internal` — inline error state in the affected view; `underlying` detail hidden behind a "show details" affordance and logged.
- `NotFound` — usually flow-controlled at the call site; if it bubbles, render inline.
- `Validation` — first blocking issue is the `message`; the full list is retrieved via `commit_pipeline.get_last_validation_issues()` and displayed in the commit panel (M4). M3 at most renders a disabled-save state with a brief summary.
- `SourceControl` / `Authentication` / `Network` — route into the recovery UI via the appropriate `ErrorState`. M3 lands the renderer scaffolding; M4 wires the full flows.
- `Conflict` — routes to the conflict resolution panel, which is strictly M4.
- `FormatVersion` — triggers the "newer version required" dialog (decision #25); this dialog lands in M3's open flow because it is a blocker to opening the project at all.
- `Cancelled` — filtered out of user-facing surfaces.

The UI **does not** catch core exceptions except at the top-level handler in `main.cpp`; `std::expected` is the conversation between core and UI.

## Spec cross-references

### §6.1 Schema Builder (verbatim)

- Users define *entity types*, each with a name and a set of typed fields
- Users also define *enum types* — named, reusable sets of labeled values (e.g., a "Status" enum with values Open, In Progress, Closed) that can be referenced by fields across any entity type
- Supported field types: text, integer, float, boolean, date/datetime, enum (references a user-defined enum type), reference (link to another entity type), and file attachment
- The `attachment` field type stores a **URI** — a URL to an externally hosted resource or a local file path. No binary data is stored inside the project directory.
- Enum values carry stable IDs, so labels can be renamed without corrupting existing records
- Schema changes are versioned; migrations are applied automatically on open
- Entity type and enum detail views display the item's UUID alongside its display name, with a one-click copy action. Because schema and record filenames are UUID-based, this is the affordance users rely on to locate a specific file in source control.

### §6.2 Record Management (verbatim)

- Create, read, update, and delete records for any entity type
- Inline editing directly in list and grid views
- Bulk import from CSV; bulk export to CSV and JSON
- Full per-record change history is provided by the source control provider's history log — no separate history mechanism is required

(Bulk import/export and source control history rendering are M4+ — M3 ships programmatic CRUD through the grid and detail view only.)

### §6.4 Views (verbatim)

- **Table view** — spreadsheet-style grid, configurable column visibility and order
- **Detail view** — single-record pane showing all fields and source control history for that record
- **Reference graph** — visual exploration of relationships between linked records (deferred to v1.1)

The detail view's "source control history for that record" is an M4 deliverable (it depends on the M4 history view). M3 ships the detail view shape with the history panel stubbed — a placeholder that reads "History available once source control UI ships (M4)." The reference graph is explicitly post-v1.

### §6.8 Change Tracking & Undo (verbatim summary)

Every user action that mutates state is a `Command`; `execute()`/`undo()` are in-memory only; the undo/redo stack is session-scoped with a commit boundary; entries at or before the boundary are rendered dimmed; the redo stack is cleared at commit. Bulk operations are wrapped in a `CommandGroup`.

### §8.3 UI Framework (verbatim)

**Decision: Qt (LGPL 3.0).** Chosen because `QTableView` with a custom model fits the record table view; Qt provides list views, form layouts, diff panels, and source control UI surfaces; the undo history is rendered as a `QListView` bound to `CommandStack::history()` — Qt renders, the custom stack owns the semantics; `QUndoStack` is deliberately not used because `CommandStack` integrates with dirty-state tracking and the `SourceControlProvider` commit lifecycle, which `QUndoStack` has no awareness of.

**Licensing: LGPL 3.0 — Dynamic Linking Required.** Dynamic linkage; installers ship the Qt libraries; EULA notifies users of their relink right; source (or pointer) available on request. Enforced from M0a in `CMakeLists.txt`.

### §8.5 UI-thread bridge (verbatim, the relevant paragraph)

The in-memory index populated by the parse workers is not the same object as the `QAbstractItemModel` rendered by the table view — the model is a UI-thread view over the index and must be mutated only from the UI thread. Workers therefore never touch any Qt object. As each worker completes its chunk, it posts a completion batch back to the UI thread via `QMetaObject::invokeMethod(target, ..., Qt::QueuedConnection)`, where `target` is a long-lived `QObject` that lives on the UI thread and holds a reference to the model. The queued invocation runs inside the Qt event loop; its handler asks the model to emit the appropriate `rowsInserted` (or `dataChanged`) signal for the newly-visible range, and surfaces any parse failure as an inline error state in the project-open UI. Batching at worker-chunk granularity — not per-file — keeps UI-thread wakeups bounded regardless of record count.

### §8.4 rule-6 exhaustiveness, constituent (2) (verbatim)

`ErrorState` → renderer coverage (lands at M3–M4). A unit test walks `enum class ErrorState`; for each value, it passes if the recovery UI has a registered renderer keyed by that value, or if the value is listed in `kUnshippedErrorStates`. Otherwise it fails. This is the second CI-enforced expression of CLAUDE.md rule 6.

## Deliverable detail

### Main window chrome (`src/ui/main_window.{hpp,cpp}`)

- Menu bar: File (New Project, Open Project, Close Project, Recent Projects, Quit); Edit (Undo, Redo — both bound to `CommandStack::can_undo()`/`can_redo()` via signal on every push); View (toggle undo history panel, toggle schema panel); Help (About).
- Title-bar provider indicator (decision #17): `"{Project Name} — {Provider display name}"`, e.g., `"Acme CRM — Git"`. Taken from a `SourceControlProvider::display_name()` accessor (naming deliberately symmetric with existing abstract accessors; the exact method name is whatever the M1a/M4 provider header exposes — if it doesn't yet, M3 adds it as a pure accessor in the interface).
- Open dialog handles decision #25's `FormatVersion` case by catching that category from `open_project()` and presenting the blocking "newer version required" modal before any UI is populated.

### Record table (`src/ui/record_table_view.{hpp,cpp}`, `record_table_model.{hpp,cpp}`)

- `RecordTableModel` subclasses `QAbstractItemModel`. Backed by `RecordManager` (M2a) and `RecordIndex` (M2b); row order comes from the active `Query` (default query = all records for the selected entity type, not deleted, ordered by `created_at`).
- Column visibility and order are per-entity-type UI state (§6.4). M3 ships the in-memory plumbing; persistence of column preferences is deferred to M5 polish.
- Inline editing: a cell edit constructs `SetFieldValue(record_id, field_id, new_value)` and pushes it through `CommandStack::push()`. The model emits `dataChanged` on push, on undo, and on redo — and on any upstream `RecordIndex` update (e.g., a record appeared from the parallel-open chunk bridge).
- Rendering 10K records at 60 fps is the §11 target. The model must not walk `RecordManager::list(...)` in `data()` — it caches the row-to-`RecordId` mapping and refreshes that mapping only on structural changes (`rowsInserted` / `rowsRemoved`).

### Record detail view (`src/ui/record_detail_view.{hpp,cpp}`)

- Displays every field of the selected record, with a widget per `FieldType`: `QLineEdit` for text, `QSpinBox`/`QDoubleSpinBox` for numerics, `QCheckBox` for boolean, `QDateEdit`/`QDateTimeEdit` for date/datetime, `QComboBox` for enum, `QComboBox` (populated from the referenced entity type's records) for reference, `QLineEdit` + browse button for attachment URI.
- Edits construct `SetFieldValue` commands — same path as the inline grid editor.
- Displays the record UUID alongside its display name with a one-click copy affordance (§6.1).
- Reserved area for "Source control history for this record" with the M4 stub text described above.

### Schema editor (`src/ui/schema_editor.{hpp,cpp}`)

- Separate panel (toggleable from View menu). Lists entity types and enums; each is edit-in-place.
- Adding / renaming / removing fields or enum values constructs schema-mutation `Command`s. These are additive M3 work — the concrete `Command` subclasses (`AddFieldDefinition`, `RemoveFieldDefinition`, `RenameEntityType`, `AddEnumValue`, `RenameEnumValue`, `HideEnumValue`, etc.) follow the same shape as M2a's three canonical commands (§7.2 worked example).
- Display UUID alongside display name with a one-click copy affordance, per §6.1.
- On commit, these changes flow through the same pipeline as record mutations, and `SchemaValidator` blocks any pre-commit change that would introduce a `Blocking` issue (§6.10, already enforced from M2a).

### Undo history view (`src/ui/command_history_view.{hpp,cpp}`, `command_history_model.{hpp,cpp}`)

- `QListView` bound to a `CommandHistoryModel` subclassing `QAbstractListModel`. The model reads `CommandStack::history()` and exposes each entry as a row.
- Entries with row index `< CommandStack::commit_boundary()` are rendered dimmed / italicised (a custom `QStyledItemDelegate` handles the styling). The boundary itself is visually distinguished (a separator row or a bold divider between pre- and post-boundary entries).
- Clicking a post-boundary row is equivalent to pressing Undo until the stack position matches; clicking a pre-boundary row is a no-op (the boundary cannot be crossed — §6.8).
- The model observes a `CommandStack::changed` signal (or equivalent callback; the exact mechanism is whatever the M2a `CommandStack` exposes — if none, M3 adds one without violating rule 2).

### Project-open UI thread bridge (`src/ui/project_open_view.{hpp,cpp}`)

- Hosts a `QObject` on the UI thread that the M2b parallel-open coordinator posts completion batches to via `QMetaObject::invokeMethod(target, ..., Qt::QueuedConnection)`.
- Each posted batch triggers a `RecordTableModel::insertRows(...)` on the row range the batch covered. Parse failures come through as `ValidationIssue` entries and are rendered as inline error banners in the table for the affected rows (not as modal errors).
- Wakeup batching stays at worker-chunk granularity per §8.5 — never per-file. The UI remains responsive throughout a 10K-record open.

### Recovery UI scaffolding and rule-6 renderer coverage (`src/ui/recovery/...`)

- `class RecoveryUiRegistry` keyed by `ErrorState`. Each registered renderer is a `QWidget` factory that builds the recovery flow's step-by-step UI from the `RecoveryFlow` struct supplied by `SourceControlProvider::get_workflow_metadata()`.
- `tests/ui/recovery_renderer_coverage_test.cpp` — the M3 expression of rule 6 constituent (2). Walks `enum class ErrorState`; for each value, asserts that either `RecoveryUiRegistry::has_renderer_for(state)` returns true or the value is in `kUnshippedErrorStates`. Iteration uses `magic_enum` (or the X-macro alternative established in M1b).
- The full per-`ErrorState` widget flows can land incrementally through M4 as the source control UI fleshes out — what must be green by end-of-M3 is the registry + the coverage test. Stub renderers ("This flow will be implemented in M4") are acceptable placeholders only if paired with an `@todo` note and a linked M4 task; the test still sees them as "covered."

### Exit criteria

- All M0a–M2b tests remain green on all three platforms (`ctest --output-on-failure`).
- `QtTest` smoke covering: (a) `RecordTableModel` round-trip — insert a record via `CreateRecord` and the model reports the new row; (b) undo/redo through the `QListView` advances and retreats correctly and redraws the boundary style.
- A demo sequence drives end-to-end: open an empty project created via the M0b CLI → schema editor builds `Customer { Name: text required, Status: enum(Open|Closed) }` → user creates ten Customer records → `CommandStack::history()` shows every mutation → user undoes three, redoes one → user commits via programmatic test hook (the M4 commit button is not shipped yet) → on-disk state matches the expected record files.
- Rule-6 renderer coverage test green on all three platforms. `kUnshippedErrorStates` still contains only `CentralizedCommitBehind`.
- Qt dynamic-linking assertion still green. `cpack` installers on all three platforms still build.

## What M4 adds next

M4 lands the full source control UI: pending changes list, pre-commit diff panel feeding from `CommandStack` dirty state, commit panel including inline rendering of the `ValidationIssue` list and the commit-disabled behaviour per §6.10 (CLAUDE.md rule 8's user-facing surface), push/pull controls using provider `sync_action_label()` / `receive_action_label()`, branch management with the branch-first default prompt (decision #18), history view rendered against `SourceControlProvider::get_history(...)`, and the conflict resolution UI rendering the three-blob `Conflict` list from M1b's `get_conflicts()`. The central `ActionId` dispatch table in `src/core/workflow.cpp` lands in M4 — rule-6 exhaustiveness constituent (3), compile-time `switch`-over-every-`ActionId` with no `default`, is enforced from M4 onward. Rule 7 (provider-aware UI guidance) finally has its full surface. The M3 per-record detail view's stubbed "Source control history" panel becomes real.
