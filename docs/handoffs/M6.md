## Mission

Ship v1.0. No new features, no perf work beyond residual bugfixing. M6 is a release engineering and hygiene milestone: version bump, final release-notes authoring, installer artefact signing and publication, deferred-item audit committed to the repo as the authoritative record of what v1 intentionally does not include, and the public-release license instrument finalised. After M6, Philotechnia is a tagged, published desktop application that installers can be downloaded for on all three target platforms.

**Predecessor:** M5 — polish, perf, installer hardening, E2E pass. **Successor:** post-v1 — deferred features (rebase-on-pull, purge-deprecated-fields, hard deletion, attachment LFS, preflight disk-space check, username rename, SVN provider, Perforce provider) pick up from an explicit v1 baseline.

## Foundation

### Project in one paragraph

Philotechnia is a cross-platform C++ desktop application for structured record management. Projects are version-controlled directories of plain JSON files (one file per record, hash-prefix directories). The application includes a first-party source control UI built on an abstract `SourceControlProvider` interface — Git via libgit2 ships in v1; SVN and Perforce are later milestones. The UI is Qt. JSON is yyjson. All undo/redo and dirty-state tracking flows through a custom `CommandStack`, not Qt's built-in undo framework.

### Non-negotiable architectural rules

Rules stay exactly as enforced through M0a–M5; M6 does not relax any of them. The release gate verifies that every CI-enforced check is green on the release build:

3. **Qt dynamically linked.** The `CMakeLists.txt` assertion is green; release installers ship Qt as shared libraries.
6. **Self-sufficient recovery.** All three rule-6 exhaustiveness constituents pass on the release build. `kUnshippedErrorStates` contains only `CentralizedCommitBehind`.
9. **Commit-pipeline-only disk I/O.** No regressions — all record writes, schema writes, open-cache writes, and (if shipped) FTS-cache writes go through the commit pipeline.

No behavioural changes to any rule occur in M6.

### Version and branding

- Version string bumped to `1.0.0` in both `vcpkg.json`'s `version-string` and the version block at the top of `CMakeLists.txt` (where CPack metadata is centralised per §15.1).
- Git tag `v1.0.0` applied once the release artefacts are signed and uploaded.
- Release notes authored under `docs/releases/v1.0.0.md` — a human-friendly summary of what v1 includes, what's intentionally deferred, and where the deferred items plug into the architecture when they ship later.

### Licensing instrument

- Decision #10 — "Source-available. Specific instrument (BSL, FSL, or custom) to be selected before first public release." — must be **resolved** by M6, not by M5's close. If M5 shipped with the instrument still unselected, M6 cannot publish.
- The selected instrument is committed in `LICENSE.md` at repo root and is the same text shipped inside every installer's EULA / bundled licenses folder.
- Qt LGPL 3.0 notice continues to ride alongside the project license — independent instruments, both visible to the user.

## Spec cross-references

### §4 Non-Goals (v1) (verbatim)

- Real-time collaborative editing (the source control provider handles asynchronous collaboration; simultaneous editing of the same record requires a commit and merge)
- A hosted/cloud-only deployment model
- A general-purpose document editor or rich text authoring tool
- Mobile clients
- Plugin or scripting system (considered for v2)

3. ~~Commit identity fallback~~ — resolved. The local profile (§5, §6.7, §13.10) is created on first launch and always provides the commit author identity. No fallback is needed.

## Deliverable detail

### Release engineering pass

- Bump `version-string` in `vcpkg.json` to `1.0.0`. Bump the CPack `PACKAGE_VERSION` block in the root `CMakeLists.txt` to match. These two edits land in the same commit.
- Re-run the full CI matrix on the release commit: every platform's build is green, every `ctest --output-on-failure` pass (unit, integration, perf harness, rule-6 exhaustiveness checks 1 and 2, and the compile-time rule-6 check 3 by virtue of compilation succeeding under `-Wswitch-enum -Werror`).
- `cpack` produces `philotechnia-1.0.0.dmg`, `philotechnia-1.0.0.msi`, `philotechnia-1.0.0.deb`, and `philotechnia-1.0.0.AppImage`. All four are signed per M5's signing infrastructure (Developer ID + notarisation stamped, Authenticode, GPG).
- Tag `v1.0.0` on the release commit once artefacts are signed.

### Release notes (`docs/releases/v1.0.0.md`)

Authored to cover, in roughly this shape:

- **What v1 includes.** One-paragraph summary mirroring the project-in-one-paragraph opener, then a bulleted feature roster pulled from §6: schema builder with nine field types; record CRUD with stateless diff-based migration; query + filter + FTS; table, detail, and schema-editor views; undo/redo via `CommandStack`; first-party source control UI over libgit2 (stage, commit, push, pull, branch, stash, history, conflict resolution); pre-commit `SchemaValidator`; mtime-based project-open cache; local-profile-only identity; OS-credential-store remote auth.
- **Supported platforms.** macOS (Apple silicon and x86-64, per the vcpkg `arm64-osx` / `x64-osx` triplets), Windows x64 (build 19041+, per the atomic-rename requirement), Linux x86-64 and arm64.
- **Explicit non-goals.** The full §4 list, restated for reader clarity.
- **Explicit deferred items.** The CLAUDE.md "Known gaps and deferred decisions" list, restated with pointers to where each will plug in when it ships. See the deferred-item audit section below.
- **Known limitations.** Network filesystem penalty (§11 warning). Attachment URI staleness (decision #22, §13.13). v1 git-only source control — SVN and Perforce interfaces exist but no provider ships.

### Deferred-item audit (`docs/known_gaps.md`)

An authoritative record of what v1 intentionally does not include, with architecture pointers for future milestones. Each entry names the gap, the owning spec section, and the architectural anchor point where the feature would plug in:

- **File attachments.** URI-only today. Future: Git LFS or content-addressed sidecar store (§6.1, §13.13, decision #22). Anchor: the `attachment` `FieldType`.
- **Hard deletion.** Soft delete only today (`deleted_at` + retained file; decision #13). Future: "compact project" destructive-confirmation action. Anchor: extend the commit pipeline to accept delete operations alongside write operations.
- **Purge deprecated fields.** `_deprecated_fields` accumulates under the stateless diff-based migration (decision #14). Future: destructive-confirmation "purge deprecated fields" action. Anchor: schema editor.
- **Field type conversion.** Blocked at commit time by `SchemaValidator` category 2 (§6.10). Future: dedicated migration flow with value mapping + preview. Anchor: schema editor's field-edit surface.
- **Rebase-on-pull.** Merge-on-pull only today (decision #23). Future: per-project or per-user rebase-on-pull option. Anchor: `SourceControlProvider::pull()` implementation behind a setting.
- **Preflight disk-space check on commit.** Commit pipeline aborts cleanly on write failure today (decision #24). Future: UX improvement estimating batch size vs. free space. Anchor: `commit_pipeline.commit(...)` early path.
- **Username rename flow.** `LocalProfile.username` treated as stable today (decision #28). Future: rename flow updating `created_by` in bulk across records. Anchor: pairs naturally with the "purge deprecated fields" action.
- **SVN provider.** Interface defined (`supports_staging()` = false, `supports_offline_commits()` = false, etc.). Candidate for v2.
- **Perforce provider.** Interface defined. Requires interface evaluation before implementation (significantly different model — changelists, mandatory checkout). Candidate for v3.
- **Plugin / scripting system.** Non-goal for v1 (§4); candidate for v2.

Each item in `docs/known_gaps.md` also carries the CI/compile-time safety net that makes it safe to defer — e.g., the `kUnshippedErrorStates` list is the on-ramp for SVN's `CentralizedCommitBehind`, and removing that entry in the SvnProvider-ships commit re-engages rule-6 check (1) for it.

### Licensing instrument finalisation

## What comes after v1.0

Not a milestone, but an orientation note for the next agent / contributor: the post-v1 work is either in the deferred list above or in the v2/v3 provider roadmap (SVN, Perforce). The architecture is explicitly designed so each deferred item plugs in at a known anchor without requiring a spec revision — the `SourceControlProvider` interface's capability flags, `WorkflowMetadata`, and `kUnshippedErrorStates` mechanisms exist precisely so new providers can land without disturbing rule 6. The `_deprecated_fields` + stateless-diff migration model means schema evolution does not require a versioned-migration system. The command pattern + commit pipeline means new user-facing actions plug into rule 9 without adding new I/O paths.

Any new work after v1.0 should start by reading `docs/known_gaps.md` and the relevant §13 decision row, then the owning §6 / §7 / §8 subsection of the spec.
