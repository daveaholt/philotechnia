## Mission

Close out everything between "feature-complete UI" and "stable v1.0 release." Fix the performance regressions that real UI interaction surfaced on top of M2b's synthetic-fixture perf numbers. Decide whether the in-memory FTS rebuild on every project open is good enough at v1 scale or whether persistence needs to ship (profile first; decide from data). Document the network-filesystem penalty clearly in user-facing docs and add whatever cheap UI accommodations reduce confusion (a warning banner on slow opens, better progress reporting). Polish the installers produced by `cpack` — code-signing on macOS, Authenticode on Windows, desktop-file + MIME associations on Linux. Run the full end-to-end distributed-collaboration flow on all three platforms as a manual test pass, with someone actually using the app. No new features ship in M5 — it is the quality-hardening milestone before M6.

**Predecessor:** M4 — full source control UI, central `ActionId` dispatch, rule-6 constituents all green. **Successor:** M6 — v1.0 stable release (code freeze, release notes, publish installers, deferred-item audit).

## Foundation

### Project in one paragraph

Philotechnia is a cross-platform C++ desktop application for structured record management. Projects are version-controlled directories of plain JSON files (one file per record, hash-prefix directories). The application includes a first-party source control UI built on an abstract `SourceControlProvider` interface — Git via libgit2 ships in v1; SVN and Perforce are later milestones. The UI is Qt. JSON is yyjson. All undo/redo and dirty-state tracking flows through a custom `CommandStack`, not Qt's built-in undo framework.

### Non-negotiable architectural rules (from CLAUDE.md)

Rules active in M5:

3. **Qt must be dynamically linked.** Installer polish must preserve this. The bundled `.dylib` / `.dll` / `.so` files are the user's copy of Qt; the installer notifies users of their right to relink, and includes a pointer to the Qt source. Code-signing the bundled Qt binaries needs to be done in a way that does not fuse them with the application binary.
5. **Hand-written ASM only with a profiled bottleneck.** M5 is when profiling is actually done. If something requires ASM or intrinsics beyond what the compiler emits, M5 is the earliest milestone a case for it can be made — and the case must be backed by a measured, reproducible regression. SIMD via `std::simd` or intrinsics is not ASM for the purposes of this rule.
9. **Commands mutate only in-memory state; disk I/O happens exclusively in the commit pipeline.** If FTS persistence lands in M5, the cache write is another call site in `src/core/commit_pipeline.cpp` — exactly like the open-cache write in M2b. Not at project open, not at project close, not as a side effect of index mutation.

Other rules are background — they were enforced earlier and M5 does not relax them. In particular, rule 4 (atomic rename) continues to govern any new cache writes; rule 6 (self-sufficient recovery) and rule 7 (provider-aware UI guidance) continue to be CI-enforced via the three exhaustiveness checks.

### Performance targets (§11, verbatim)

| Metric | Target | Notes |
|--------|--------|-------|
| Cold start time | < 500 ms on reference hardware | Excludes project open time |
| Project open (10K records, local SSD) | < 3 seconds | Directory walk + per-file JSON parse; parallel async I/O should be used to mitigate syscall overhead |
| Project open (subsequent opens) | < 1 second | mtime-based cache skips unchanged files; only modified files are re-parsed |
| Record list render (10,000 records) | < 16 ms per frame (60 fps) | Operates on in-memory index, not disk |
| Full-text search response | < 500 ms on 100K text field values | Operates on in-memory index |
| Write throughput | ≥ 500 record writes/second (single writer) | One JSON file per record + atomic rename per write |

**Network filesystem warning (verbatim).** Projects opened from a network-mounted path (NFS, SMB/CIFS, network drives) will have significantly higher open times due to per-file I/O latency. 10K files over a typical LAN could take 30–60+ seconds on first open. This is a known limitation of the one-file-per-record design and should be communicated clearly in the documentation. The recommended practice is to keep the working copy on local storage and use the source control provider for sharing.

### Security posture (§12, verbatim)

- No application-level passwords. Identity is based on the local profile (username + email stored in OS app data), which supplies commit author information to the source control provider. No credentials, user registry, or role data are stored in the project repo. Access control, where required, is enforced at the VCS hosting level (branch protection rules, repository permissions, signed commits) — Philotechnia does not implement application-layer permissions and does not attempt to.
- No at-rest encryption in v1. Projects are plain JSON on disk. Users needing confidentiality against filesystem-level access should rely on OS-level full-disk encryption (FileVault, BitLocker, LUKS), which does not interfere with the source control provider's ability to diff and merge files.
- No telemetry, no network calls unless a push or pull is explicitly initiated by the user.
- Commit authorship is tied to the logged-in Philotechnia user via the source control provider, providing a tamper-evident audit trail through the provider's history.
- Remote credentials are managed by the host OS credential store where supported (git credential helpers on all three platforms).
- The project-open cache (§8.6) stores raw record JSON under the OS app data directory. This mirrors the content of the project files on disk — same trust model — and does not introduce a new confidentiality risk, but is worth noting when describing the on-disk footprint to users on a multi-user machine.

## Spec cross-references

### §8.5 Async I/O, platform-native deferral note (verbatim)

**Platform async I/O:** For the v1 implementation, `std::async` / `std::future` with a manually managed thread pool is sufficient and avoids introducing a platform-specific I/O framework dependency. If profiling reveals that syscall overhead (rather than parse time) is the bottleneck at scale, platform-native async I/O (Linux `io_uring`, Windows IOCP, macOS `kqueue`) can be added as a future optimisation behind an abstraction layer. This is deferred until profiled evidence justifies the complexity.

M5 is the milestone where the deferral expires: if profiling **does** show syscall overhead is the bottleneck, the work either ships in M5 or explicitly moves to post-v1 with a documented "deferred" note. No ASM/SIMD/`io_uring` work goes in without profiled evidence.

### §8.6 Project-open cache (verbatim summary, lifecycle)

The cache is written only after a successful commit — the natural checkpoint where in-memory state matches on-disk state. Open does not write the cache; project close does not write the cache. This extends CLAUDE.md rule 9: the commit pipeline is the only code path that writes record, schema, or cache files.

If M5 ships FTS persistence, it follows the same discipline — write on commit only, invalidate on schema fingerprint or `cache_format_version` change.

### §15.6 M2b exit — FTS persistence explicitly deferred (verbatim)

> **Persistence is deliberately not part of M2b.** The FTS index is rebuilt from records on project open. Persisting the index is a future perf milestone if profiling shows the rebuild is a bottleneck at 100K values; at M2b, the §11 target of <500 ms for a full-text query over 100K values is met with an in-memory rebuild that runs as part of the parallel open walk.

M5 is the milestone that future refers to. The decision to ship FTS persistence is strictly profile-driven — no speculative work.

## Deliverable detail

### Profiling pass across all three platforms

- Re-run the M2b `tests/perf/` harness under the real UI stack. Compare cold-open, warm-open, record-list-render-per-frame, FTS-query-latency, and write-throughput numbers against the §11 targets.
- Instrument a realistic interactive session on each platform (macOS Instruments Time Profiler, Windows VS Profiler / ETW, Linux `perf record` + `hotspot`) covering: open a 10K-record project → scroll through the table → edit 50 fields across 20 records → commit → push → pull with a conflict → resolve → commit merge.
- Produce a short write-up per platform with: top-N hotspots, whether each is a §11-regression vs. M2b baseline, and a recommended action (fix, defer, accept). Commit the write-ups under `docs/perf/m5_<platform>.md` so the evidence is preserved.

### Hotspot fixes

- Any fix has a microbenchmark that reproduces the speedup. No speculative optimisation — profile, fix, re-profile.
- If fixes introduce ASM or intrinsics, they must pass CLAUDE.md rule 5: reproducible regression, clear before/after measurement, narrowly scoped translation unit. SIMD via `std::simd` / intrinsics is fine without ceremony; hand-ASM is not.

### FTS persistence (conditional)

- **Ships only if profiling shows the rebuild is a bottleneck at 100K values.** Criterion: cold-open time attributable to FTS rebuild exceeds 500 ms on reference hardware in steady state.
- If shipped:
  - Cache layout per `EntityTypeId` under the same OS-app-data cache directory used by the M2b open cache, keyed by `project.id`.
  - Invalidation: per-entry on mtime/size mismatch (same rationale as §8.6); whole-cache on schema fingerprint change, `cache_format_version` change, or corrupt-on-read.
  - Written only from the commit pipeline — one additional call site in `src/core/commit_pipeline.cpp` after the open-cache write, using `atomic_write`.
  - Rebuild-from-disk stays as the canonical fallback; the cache is a hit path, not a source of truth.
- If not shipped: document the decision in the perf write-up and explicitly list "FTS persistence" as a post-v1 candidate in M6's deferred-item audit.

### Network-filesystem documentation and UX

- README section + user-facing docs page explaining the one-file-per-record design's interaction with NFS / SMB / network drives, with the specific 30–60+ seconds figure from §11.
- **Slow-open banner:** if cold open exceeds 10 seconds, show a non-modal banner in the project-open UI: *"This project is opening slowly — it may be on a network drive. Consider keeping the working copy on local storage. [Learn more]"*. Cheap to implement; noticeable when it helps.
- Optional polish (profile to decide): a progress indicator that updates per worker batch (the M2b bridge already has the wakeup granularity needed), so users see per-record feedback during long opens.

### Installer polish

- **macOS `.dmg`:** Signed with a Developer ID Application certificate; notarised via Apple's notary service; stapled so Gatekeeper is silent on open. Drag-to-Applications background image. `Info.plist` carries a proper app icon (`.icns`), bundle identifier, supported document types. LGPL 3.0 notice in About.
- **Windows `.msi`:** Signed with Authenticode (EV cert if available for SmartScreen reputation). Start Menu shortcut, file association for `manifest.json` to "open as Philotechnia project," proper add/remove programs metadata. Uninstaller removes app files but preserves the OS-app-data cache (per decision #26's 90-day orphan sweep policy — uninstall is not a cache-clear event). LGPL 3.0 notice in the EULA.
- **Linux `.deb` + `.AppImage`:** Desktop entry with MIME type association, icons at the required sizes, AppStream metadata file. `.deb` uses standard postinst/postrm scripts; `.AppImage` runs without install and is self-contained except for the LGPL Qt binaries shipped alongside. LGPL 3.0 notice in a bundled `LICENSES/` directory.
- Qt LGPL 3.0 compliance on every installer: bundled Qt libraries are the user's copy; installer notifies users of their right to relink; source pointer (link to Qt's source repo at the pinned version) included in the bundle.
- Licensing instrument (decision #10): if the BSL / FSL / custom-license decision has not been made by M5 kickoff, make it here. The installer's EULA cannot ship without it.

### End-to-end testing across all three platforms

- Manual test pass. A person goes through the distributed-team workflow on each platform:
  1. Fresh OS install (or VM), install from the M5 installer.
  2. First-launch: create local profile, open an empty project, create schema, add 20 records.
  3. Second machine (different platform): clone from a shared remote (GitHub, GitLab, or Gitea per decision #21), open the project, set local profile, add 10 records on a working branch, push.
  4. First machine: pull. Observe the branch-first prompt, the merge, the updated record table.
  5. Both machines edit the same record's same field; one pushes, the other pulls and resolves the conflict through the UI.
  6. Primary-branch protection (branch-first default) observed: attempting to edit on `main` prompts to create a working branch.
  7. Recovery flows exercised deliberately: detach HEAD via an external `git checkout <sha>`, reopen in Philotechnia, use the `DetachedHead` recovery flow to return to a branch. Similarly: force a push rejection, force an authentication failure, force a remote-unreachable state.
- Findings get logged as M5 issues and fixed. None shipping to M6.

### Exit criteria

- Every §11 target still passes on the CI perf harness **and** in a real UI interactive session on all three platforms. Any regressions introduced by M3/M4 UI work are fixed.
- Documented perf write-ups exist per platform under `docs/perf/`.
- If FTS persistence shipped: all §8.6-analogous invalidation cases pass tests; warm-open time improves measurably on the 100K-FTS-value fixture.
- If FTS persistence did not ship: a documented decision note in `docs/perf/` with the supporting measurements.
- Installer on every platform: signed, minimal warnings at install time, passes OS-native integrity checks (Gatekeeper, SmartScreen, Linux desktop-file validation).
- Manual E2E test pass on all three platforms; every §6.6 and §6.9 flow works; every recovery flow returns the user to a clean state without external tools (rule 6 observable).
- Licensing instrument finalised (decision #10). EULA ships with the correct text.
- All M0a–M4 CI still green.

## What M6 adds next

M6 is the release milestone — no new features, minimal new code. Version bumped to 1.0.0, release notes, installer artefacts signed and published, a final audit of `docs/known_gaps.md` (or the equivalent) to record what is deferred to post-v1, and the flip of `"version-string"` in `vcpkg.json` + release tags. Any remaining deferred items (rebase-on-pull, purge-deprecated-fields, hard deletion, LFS-style attachments, preflight disk-space check, rename-username flow) are explicitly listed as post-v1, with pointers to where they will plug into the architecture when they ship. SVN and Perforce providers remain interface-only.
