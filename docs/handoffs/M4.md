## Mission

Make Philotechnia a complete collaborative tool by wrapping `SourceControlProvider` (M1a) in the full first-party source control UI, with every surface the spec requires: pending changes list, pre-commit diff panel, commit panel that renders every `ValidationIssue` inline and disables commit until every `Blocking` issue is resolved (CLAUDE.md rule 8's user-facing surface), push/pull controls labelled by provider, branch management with the branch-first default prompt, history view, conflict resolution UI that presents each `Conflict` field-by-field (never as raw markers), and the post-pull invalid-state banner + resolution flows for state that arrives from outside the application. The central `ActionId` dispatch table in `src/core/workflow.cpp` lands here, carrying rule-6 exhaustiveness constituent (3) — a compile-time `switch`-over-every-`ActionId` with no `default` case, so adding an `ActionId` without a dispatch entry becomes a build failure. Constituent (2) — `ErrorState` → renderer coverage — closes out in M4 if M3 did not already finish it. After M4, a user has never needed to open a terminal or an external VCS client to resolve any state Philotechnia created (CLAUDE.md rule 6 is fully observable behaviourally).

**Predecessor:** M3 — UI alpha (main window, record table, detail view, schema editor, undo history widget, recovery-UI scaffolding). **Successor:** M5 — polish and performance (profiling hotspots from M3/M4 UI work, FTS persistence if needed, installer polish, network-filesystem accommodations).

## Foundation

### Project in one paragraph

Philotechnia is a cross-platform C++ desktop application for structured record management. Projects are version-controlled directories of plain JSON files (one file per record, hash-prefix directories). The application includes a first-party source control UI built on an abstract `SourceControlProvider` interface — Git via libgit2 ships in v1; SVN and Perforce are later milestones. The UI is Qt. JSON is yyjson. All undo/redo and dirty-state tracking flows through a custom `CommandStack`, not Qt's built-in undo framework.

### Non-negotiable architectural rules (from CLAUDE.md)

Rules active in M4:

1. **The UI layer must never depend on a concrete `SourceControlProvider` implementation.** The central load-bearing rule in M4. Every M4 UI widget takes a `SourceControlProvider&` (abstract) — never a `GitProvider&`. Action labels come from `sync_action_label()`, `receive_action_label()`, `commit_action_label()`. Branching widgets hide themselves based on `supports_branching()`, not on `provider.is_git()`. The dispatch table is the only indirection that knows how to translate an `ActionId` into a concrete method call — UI widgets emit `ActionId` values and never `switch` on them.
6. **The application must be fully self-sufficient for all source control states it creates.** M4 finishes the story: every `ErrorState` has a renderer (constituent 2 green at the end of M4 at the latest), every `ActionId` has a dispatch entry (constituent 3 enforced as a compile-time `switch`). The user never needs an external tool to exit a Philotechnia-created state.
7. **The source control UI is provider-aware and guides users toward correct workflows.** M4 is where rule 7 becomes visible. Provider-supplied `WorkflowMetadata.branching_model_description` is shown in orientation UI. Recovery flows render from provider-authored step text. The branch-first default (decision #18) prompts on any edit to the primary branch unless `supports_branching()` returns false.
8. **Schema invariants are enforced at commit time, not discovered at pull time.** `SchemaValidator` still runs in the commit pipeline (M2a); M4 surfaces its output. Every `ValidationIssue` in `commit_pipeline.get_last_validation_issues()` is rendered inline in the commit panel; every `Blocking` issue disables the commit action until resolved.
9. **Commands mutate only in-memory state; disk I/O happens exclusively in the commit pipeline.** Source control operations (push, pull, stash, branch switch) are not `Command`s — they act on the provider, not on the in-memory project state, and their side effects are observed through `get_pending_changes()`, `get_conflicts()`, and the commit pipeline. Any UI flow that appears to need a `Command` for a source-control-side side effect is a design error — it should be a direct provider call plus a state refresh.

Rules 2, 3, 4, 5 are background: rule 2 (CommandStack ownership) — undo history continues to work from M3's binding; rule 3 (Qt dynamic) enforced from M0a; rule 4 (atomic rename) already governs every record and cache write in the commit pipeline; rule 5 (no ASM) passively observed.

### Storage format (§7.1 summary)

Project = `manifest.json` + `schema/...` + `records/{entity-type-id}/{xx}/{record-uuid}.json`. M4's commit panel feeds from `CommandStack::dirty_records()` and `provider.get_pending_changes()` — the two sources, reconciled. Pending changes surfaced independent of in-memory state (§6.9): on-disk changes from an aborted batch, an external JSON edit, or a prior session are always visible.

### Error type (§8.7 summary)

`Error::Category` — `SourceControl`, `Authentication`, `Network`, `Conflict`, `Validation` are the M4-relevant categories. `SourceControl`/`Authentication`/`Network` route into the recovery UI via the appropriate `ErrorState`; `Conflict` routes to the conflict resolution panel; `Validation` carries the first blocking issue as `message`, the full list via `commit_pipeline.get_last_validation_issues()`.

## Spec cross-references

### §6.6 Collaboration & Sync (verbatim)

The configured source control provider is the collaboration mechanism. There is no custom sync protocol or server.

- **Sharing:** The project directory is pushed to a shared remote. Team members clone or check out the project locally and open it in Philotechnia.
- **Committing:** Changes accumulate in the working state (dirty state tracking, §6.8) until the user explicitly commits via the source control panel. The logged-in Philotechnia user is mapped to the commit author identity. For centralized providers, committing requires network connectivity.
- **Pushing and pulling:** Initiated from the source control panel. Pull fetches and merges remote changes; conflicts are surfaced in the conflict resolution UI (§6.9) rather than as raw file-level conflict markers.
- **Branching:** Where the provider supports it, users can create and switch branches from the source control panel.

### §6.9 Source Control Integration (verbatim)

The source control UI **guides users toward correct workflows** — it does not merely expose VCS operations. Where the active provider supports branching, the application enforces a **branch-first default**: if the user is on the primary branch and makes any change, the app prompts them to create or select a working branch before proceeding. The user can dismiss the prompt and continue on the primary branch, but branch-first is the strongly encouraged default path. Multiple branches can be open simultaneously; the user can switch between them from the source control panel.

**Stashing** is supported for providers where `supports_stashing()` is true. If a user has uncommitted changes on a branch and needs to switch context, they can stash their work, switch branches, and return to it later. The stash list is visible in the source control panel.

**Primary branch** is configured at project creation time — the user confirms the name (git default: "main"; SVN default: "trunk"; provider suggests via `suggested_primary_branch_name()`). The primary branch name is stored in `manifest.json`. For projects opened from an existing repository that predates Philotechnia, the app detects the current default branch and prompts the user to confirm it before the first session begins.

Workflow language and guidance text are provider-appropriate: git's branch model is described differently from SVN's copy-based branches or Perforce's streams. Each provider contributes a `WorkflowMetadata` struct (via `get_workflow_metadata()`) that supplies a branching model description and a set of error recovery flows — so the UI can render provider-aware guidance without knowing the concrete provider type (see §13).

**Hard quality bar:** Philotechnia must be fully self-sufficient for every source control state it can create. If the application can put a repository into a state — conflict, detached HEAD, incomplete merge, lock, authentication challenge — it must be able to guide the user out of that state within the application. No user should ever need to open a terminal or an external VCS client to resolve a condition Philotechnia created.

The source control panel provides:

- **Pending changes list** — all modified, added, and deleted record and schema files since the last commit, described in human-readable terms ("Record 'Acme Corp' modified", "Entity type 'Invoice' added") rather than raw file paths. Pending changes are surfaced independent of the in-memory `CommandStack` state — so on-disk changes from an aborted commit batch, an external JSON edit, or a prior session are always visible and never silently invisible to the user
- **Pre-commit diff panel** — field-by-field breakdown of pending changes per record; ability to revert individual field changes before committing; feeds directly from `CommandStack` dirty state
- **Commit panel** — write a commit message and commit pending changes; Philotechnia user identity is mapped to the provider's author concept; disabled without connectivity for centralized providers; disabled while any `Blocking` pre-commit validation issue is unresolved (see §6.10)
- **Push / pull controls** — labelled using the provider's `sync_action_label()` and `receive_action_label()` methods; remote status indicator drawn from `RemoteStatus`
- **Branch management** — create, switch, and delete branches; hidden for providers where `supports_branching()` returns false
- **History view** — commit log rendered as a timeline of human-readable change summaries; selecting a commit shows a record-level diff
- **Conflict resolution UI** — when a sync produces conflicts, affected records are shown side-by-side (local vs. incoming), field by field, with options to accept either version or manually set a value; raw file-level conflict markers are never exposed to the user.

### §6.10 Pre-commit Validation — UI surfacing (verbatim summary)

`SchemaValidator` runs in the commit pipeline. If validation fails, the commit panel surfaces the issues inline and the commit action is disabled until every `Blocking` issue is resolved. Each `ValidationIssue` carries a `category`, `severity`, `message`, `affected_ids`, and optional `suggested_fix`. The UI renders these in the commit panel; the user resolves them through the normal schema editor, record editor, or a bulk-fill flow, then re-triggers validation. `Warning` issues are shown but do not block. **Post-pull invalid state** (the defense-in-depth case) is flagged with a banner in the UI and offered the same resolution flows, but does not block editing or further commits on other records.

### §8.4 — Full `SourceControlProvider` interface, `ActionId`, dispatch rule, constituents 2 and 3 (verbatim)

```cpp
enum class ActionId {
    Retry, Cancel,
    Pull, Push, Sync,
    Stash, ApplyStash, DropStash,
    AbortMerge, ContinueMerge, OpenConflictResolver,
    ReattachHead, CreateBranchHere,
    Reauthenticate, EditCredentials,
    OpenSettings,
};
```

**Action dispatch (verbatim).** `ActionId` values resolve to behaviour through a single dispatch table in `src/core/workflow.cpp`. UI widgets never `switch` on `ActionId` themselves — they emit the value to the dispatcher, which invokes the right combination of `SourceControlProvider` method, application command, or UI navigation. This keeps the UI layer free of any knowledge about how recovery actions actually execute (and preserves CLAUDE.md rule 1, which forbids UI from depending on a concrete provider).

**Rule-6 exhaustiveness constituents (2) and (3) (verbatim).**

2. **`ErrorState` → renderer coverage (lands at M3–M4).** Once the recovery UI exists, a second test walks the enum and asserts that the recovery-UI renderer handles every value, with the same `kUnshippedErrorStates` exemption applied.
3. **`ActionId` → dispatch entry coverage (lands at M4 with the dispatch table).** This is a compile-time exhaustiveness check: the dispatch function in `src/core/workflow.cpp` is a `switch` over every `ActionId` enumerator with no `default` case, so adding an `ActionId` without a dispatch entry is a build failure rather than a CI failure.

**Resolution semantics (verbatim summary).** `resolve()` is called once per conflicted file after the application core has reconciled the three blobs surfaced by `get_conflicts()` into a user-approved result. The `resolved_content` parameter is the complete post-merge contents of the file as UTF-8 bytes — for record and schema files this is the fully merged JSON. A present value instructs the provider to write the file (via atomic temp-file-then-rename) and clear the conflict from its index; `std::nullopt` instructs the provider to accept a delete. The provider does not produce a merge commit itself; the application invokes the normal commit pipeline after every conflict has been resolved. Partial resolution is supported; `commit()` fails with `Category::Conflict` if invoked while conflicts remain.

## Deliverable detail

### Source control panel shell (`src/ui/source_control/source_control_panel.{hpp,cpp}`)

- Dockable panel in the main window. Contains the pending changes list, pre-commit diff panel, commit panel, push/pull controls, branch management, history view, and (when active) the conflict resolution panel and post-pull invalid-state banner.
- Holds a reference to the `SourceControlProvider&` active for the current project. Capability flags (`supports_staging()`, `supports_offline_commits()`, `supports_branching()`, `supports_stashing()`) gate which sub-panels are visible.
- Connectivity indicator (decision #20): always visible. Calls `provider.remote_status()` periodically (or on relevant events) and displays reachability; for centralized providers, drives the commit-disabled-when-offline behaviour.

### Pending changes list (`src/ui/source_control/pending_changes_view.{hpp,cpp}`)

- Backed by a `QListView` over a `QAbstractListModel` adapter that reads `provider.get_pending_changes()` on demand.
- Human-readable entries per §6.9 — `PendingChange::description` is the text; `file_path` is surfaced in a "show file path" affordance.
- Refreshes on: user commit success, pull, external pending-change notification (if the M2b open pipeline exposes one), and manual "Refresh" action.

### Pre-commit diff panel (`src/ui/source_control/pre_commit_diff_panel.{hpp,cpp}`)

- Field-by-field breakdown per dirty record, sourced from `RecordWorkingCopy::diff()` (M2a).
- Each row: field name, baseline value, working value. "Revert this field" affordance constructs an inverse command (or directly revises the working copy through a dedicated `Command` subclass — whichever keeps rule 9 clean).
- Selection here interacts with the main table: selecting a dirty record highlights the matching row in the record table.

### Commit panel (`src/ui/source_control/commit_panel.{hpp,cpp}`) — **rule 8 user-facing surface**

- Multiline text input for the commit message, sized for a subject line + body convention.
- Commit button label = `provider.commit_action_label()`. Disabled when:
  - No dirty records or schema changes (nothing to commit).
  - `supports_offline_commits() == false` and the connectivity indicator says offline (decision #20).
  - Any `ValidationIssue` with `severity == Blocking` is present in `commit_pipeline.get_last_validation_issues()` (decision #15, CLAUDE.md rule 8).
- Validation issue list rendered inline above the button. Each issue:
  - Shows the category badge (SchemaIntegrity / SchemaChangeCompat / RecordIntegrity / ReferentialIntegrity), severity icon, and message.
  - Shows `suggested_fix` text if present.
  - If `affected_ids` has schema-element UUIDs, clicking jumps to the schema editor scoped to that element; if it has record UUIDs, clicking jumps to the record table filtered to those rows.
- On commit action: invoke `commit_pipeline.commit(...)` (M2a). On success, refresh pending changes; on `Error::category == Validation`, re-render the issue list (post-edit validator re-runs happen continuously; this path is the fallback); on `Error::category == Conflict`, route to the conflict resolution panel; on any other `Error`, route through the recovery UI.

### Push / pull controls (`src/ui/source_control/sync_controls.{hpp,cpp}`)

- Push button label = `provider.sync_action_label()`; pull button label = `provider.receive_action_label()`.
- Remote status indicator drawn from `RemoteStatus`: `local_only` and `remote_only` counts are displayed; `available == false` renders a muted "remote unreachable" label and disables both buttons with inline explanation.
- Successful `pull()` that encounters conflicts routes to the conflict resolution panel (see below). Successful `pull()` that introduces invalid schema state (post-pull validator backstop) shows the invalid-state banner.

### Branch management (`src/ui/source_control/branch_manager.{hpp,cpp}`)

- Hidden entirely when `provider.supports_branching() == false`.
- Lists `get_branches()`; shows `current_branch()` prominently.
- Create / switch / delete actions. Switch dispatches through the `ActionId` dispatcher; if the switch fails with `DirtyWorkingCopyOnBranchSwitch`, routes to the corresponding recovery flow (stash → switch → apply stash, per M1b's `RecoveryFlow` wiring).
- **Branch-first default (decision #18):** a user on the primary branch who pushes any `Command` through `CommandStack` triggers a modal prompting them to create or select a working branch. Dismissible; state per session so users are not re-prompted on every edit after dismissal. The prompt is universal — no operation-specific trigger list.
- **Primary-branch confirmation on first open of a pre-existing repo:** if `manifest.json.primary_branch` is unset and the repo has commits, detect the current default and prompt the user to confirm before the first session begins. Writes the confirmed name to `manifest.json` via the commit pipeline on the next commit.
- **Stash UI:** visible when `supports_stashing() == true`. Displays `get_stashes()`; supports `stash()`, `apply_stash()`, `drop_stash()`.

### History view (`src/ui/source_control/history_view.{hpp,cpp}`)

- Reads `provider.get_history(file, limit)` — scoped to the selected record file for the record detail view's now-live history panel, or to the whole project for the panel-level view.
- Timeline rendering with `HistoryEntry.{id, author, message, timestamp}`. Selecting a commit shows a record-level diff (read from the provider or reconstructed from stored baselines; M4 picks whichever is feasible given the provider surface — for git, `git_diff_tree_to_tree` works).
- The M3 stub panel in the record detail view ("History available once source control UI ships (M4)") is replaced with this view scoped to the single record.

### Conflict resolution UI (`src/ui/source_control/conflict_resolver/...`)

- Consumes `provider.get_conflicts()` → three-blob list per path.
- JSON-aware field-level merger (the §6.9 "lives in the application core, not in the provider" logic) takes the three blobs and produces a `FieldLevelConflictView` per affected record: for each field, show ancestor / local / incoming / merged-so-far. Reuses M3's per-`FieldType` widgets for editing the merged value.
- Per-conflict actions: "Accept local", "Accept incoming", "Manually edit". On completion, call `provider.resolve(path, merged_content)`; for delete-vs-modified with user choosing the delete, call `provider.resolve(path, std::nullopt)`. Partial resolution is supported: the user can close the panel mid-resolution and `get_conflicts()` continues to report the unresolved set.
- When the last conflict is resolved, the commit panel's merge-commit path runs — the same `commit_pipeline` that handles ordinary commits, producing a merge-commit message. `commit()` fails with `Category::Conflict` if invoked while any conflict remains.

### Post-pull invalid-state banner (`src/ui/source_control/invalid_state_banner.{hpp,cpp}`)

- The defense-in-depth surface for §6.10's post-pull case. Invalid state (from external JSON edits, older Philotechnia versions, or tooling outside the app) is detected by running `SchemaValidator::validate(...)` after any `pull()` completes.
- Shows a non-modal banner at the top of the record table: `"N records have schema issues from a recent sync. Review and resolve."` Clicking offers the same resolution flows as the commit panel (schema editor, record editor, bulk fill).
- Does **not** block editing or further commits on other records — only blocks commits that involve affected records via the standard pre-commit validator path.

### Central `ActionId` dispatch (`src/core/workflow.cpp`)

- `dispatch(ActionId, DispatchContext&) -> std::expected<void, Error>` — a plain `switch` over every `ActionId` enumerator with **no `default` case**.
- `DispatchContext` bundles the references the dispatch entries need: `SourceControlProvider&`, `CommandStack&`, `UiNavigator&` (for recovery flows that route to a different panel).
- Entries by action, non-exhaustively:
  - `Retry` — re-run the last attempted operation (context supplies a retry closure).
  - `Cancel` — dismiss the recovery flow, no-op.
  - `Pull` / `Push` / `Sync` — call `provider.pull()` / `provider.push()` / the centralized `commit()` path.
  - `Stash` / `ApplyStash` / `DropStash` — call the matching provider method.
  - `AbortMerge` / `ContinueMerge` / `OpenConflictResolver` — recovery actions for `IncompleteMerge`; `OpenConflictResolver` is navigation, the other two are provider calls through wrappers that stay rule-1-clean.
  - `ReattachHead` / `CreateBranchHere` — recovery actions for `DetachedHead`; call `switch_branch` / `create_branch` as appropriate.
  - `Reauthenticate` / `EditCredentials` — recovery actions for `AuthenticationRequired`; drive the credential UI + OS credential store.
  - `OpenSettings` — navigation to the settings dialog.
- **Constituent (3) of rule 6 — compile-time exhaustiveness.** Because the `switch` has no `default`, a new `ActionId` value added without a new `case` entry fails to compile under `-Wswitch-enum -Werror`. The project CMake already sets this on all three platforms; M4 verifies it is still in effect and adds a documented comment in `workflow.cpp` explaining the discipline.

### Rule 6 constituent (2) close-out

- If M3 left the renderer coverage test stubbed, M4 completes the per-`ErrorState` renderers as the source control UI flows out. `CentralizedCommitBehind` stays in `kUnshippedErrorStates`; every other `ErrorState` has a real renderer by end of M4.

### Exit criteria

- A user can, from a stock checkout of the project, complete the end-to-end "distributed team member" flow: clone → open → set local profile → create records on a working branch → push → another machine pulls → edits → pushes → first machine pulls with a deliberate conflict → resolves through the conflict UI → commits merge → pushes — all without opening a terminal (CLAUDE.md rule 6 observable).
- Every `ValidationIssue` category (SchemaIntegrity, SchemaChangeCompat, RecordIntegrity, ReferentialIntegrity) has at least one fixture-driven UI test proving the commit button disables and the issue renders inline.
- Rule-6 exhaustiveness: constituents (1), (2), and (3) all green. `kUnshippedErrorStates` still contains only `CentralizedCommitBehind`.
- Decision #20 accommodation visible: a mock centralized provider (possibly a variant of `TestProvider` from §8.4a) drives the commit-disabled-when-offline scenario in a test.
- All M0a–M3 tests remain green on all three platforms; M4 adds UI tests through `QtTest` for the commit panel's `ValidationIssue` rendering, the branch-first prompt, and the conflict resolver's field-level UI.
- Qt dynamic-linking assertion still green. `cpack` installers on all three platforms still build and include every new UI surface.

## What M5 adds next

M5 is the polish milestone. It picks up the perf hotspots that UI work in M3 and M4 surfaced (the §11 targets were met at M2b on synthetic fixtures; M5 closes any regressions that real UI interaction introduced), evaluates whether the M2b in-memory FTS rebuild on every project open is acceptable at the 100K-value envelope or whether persistence is needed, documents and (where cheap) accommodates the network-filesystem penalty called out in §11, and finalises the installer polish for all three platforms. It is also the last milestone before v1.0 stability in M6 — any remaining "nice to have" items from the §13 known gaps get scoped here or pushed explicitly to post-v1.
