# Resolved Decisions — Philotechnia

Quick-reference for settled architectural choices. For the full rationale behind each decision, see the relevant section of [`philotechnia_spec.md`](philotechnia_spec.md).

---

| # | Decision | Choice | Spec ref |
|---|----------|--------|----------|
| 1 | **Sync / collaboration architecture** | `SourceControlProvider` abstract interface. Git (libgit2) ships in v1. No custom CRDT or sync server. Real-time sync is a non-goal. | §8.4, §13.1 |
| 2 | **Storage format** | One JSON file per record, in hash-prefix directories (`records/{entity-type-id}/{xx}/{uuid}.json`). Human-readable, VCS-agnostic. All filenames (records, entity types, enums) use UUIDs rather than slugs for uniformity and to avoid rename/collision handling; the UI surfaces UUIDs alongside display names so users can locate files in git. | §7.1, §13 |
| 3 | **JSON library** | yyjson — fast for both read and write, C99, single dependency. | §9, §13.6 |
| 4 | **UI framework** | Qt 6 (LGPL 3.0). Must be dynamically linked. `QTableView` for record grids; `QListView` bound to `CommandStack::history()` for undo rendering. `QUndoStack` is not used. | §8.3, §13.7 |
| 5 | **Undo/redo architecture** | Custom `CommandStack` in the application core — not `QUndoStack`. Required because `CommandStack` integrates with `SourceControlProvider` commit lifecycle and dirty state tracking. | §7.2, §8.3 |
| 6 | **Source control abstraction** | Strategy pattern via `SourceControlProvider` abstract base class. UI and core depend only on the interface. Provider capability flags (`supports_staging()`, `supports_offline_commits()`, `supports_branching()`) drive UI feature visibility. | §8.4 |
| 7 | **Assembly / low-level code** | C++23 is the default. Hand-written ASM requires a profiled, reproducible bottleneck to justify it. | §9, §13.3 |
| 8 | **Installer strategy** | CMake + CPack. Artifacts: `.dmg` (macOS), `.msi` (Windows), `.deb` / `.AppImage` (Linux). | §13.4 |
| 9 | **Large dataset ceiling** | 50,000 records per entity type for v1. Hash-prefix structure keeps filesystem and VCS index healthy at this scale. Sparse checkout / partial clone to be evaluated per-provider beyond this scale. | §13.5 |
| 10 | **Licensing** | Source-available. Specific instrument (BSL, FSL, or custom) to be selected before first public release. | §13.2 |
| 11 | **Offline-first scope** | "Offline-first" applies to distributed providers (git) only. Centralized providers (SVN, Perforce) require connectivity for commits — this is a known constraint communicated in the UI. | §3 |
| 12 | **No application-layer access control** | Philotechnia has no project-level user registry and no role system (no `users.json`, no Owner/Editor/Viewer). Every team member who can open the project is treated identically by the application. Access control, where required, is enforced at the VCS hosting level — branch protection, repository permissions, signed commits. The project is a plain directory of JSON files under version control; any application-layer role check would be trivially bypassable, so pushing the security boundary to the VCS host is the only model that produces a real guarantee. | §5, §6.7 |
| 13 | **Soft deletion** | Records are soft-deleted (`deleted_at` field retained on disk). Hard deletion (file removal) is deferred to a future "compact project" action to preserve per-file VCS history. | §7.2 |
| 14 | **Schema migration strategy** | Stateless diff-based — not step-versioned. Schema files are the migration target; reconciliation runs in-memory at record load time; disk is not touched until the normal commit pipeline. Missing fields receive declared defaults; removed fields move to `_deprecated_fields` on the record. No `schema_version` field in `manifest.json` — schema files are the authoritative state; a fingerprint over them is computed when a cache key is needed. A "purge deprecated fields" action (surfaced as a destructive confirmation dialog) is required for hard removal. | §7.2, §13.15 |
| 15 | **Pre-commit validation** | `SchemaValidator` runs in the commit pipeline and blocks commits that would produce invalid state. Four categories: schema internal consistency, schema change compatibility, record integrity, referential integrity. Check logic + tests land in M2; commit-panel UI surfacing lands in M4. Post-pull invalid state is the backstop case — flagged and resolved incrementally, does not block editing. | §6.10, §7.2 |
| 16 | **Async I/O approach** | `std::async` thread pool for v1 project open (parallelise directory walk + JSON parse). Platform-native I/O (io_uring / IOCP / kqueue) is a deferred optimisation pending profiling evidence. | §8.5 |
| 17 | **Provider selection UX** | Auto-detect provider silently on open (no confirmation dialog). Active provider displayed persistently in the application title bar. | §13.8 |
| 18 | **Branch strategy UX** | Branch-first strong default. Any change on the primary branch triggers a prompt to create or select a working branch. Prompt is dismissible but branching is the default path. Multiple branches can be open simultaneously. Stashing supported where `supports_stashing()` == true. Primary branch name configured at project creation (provider suggests via `suggested_primary_branch_name()`), stored in `manifest.json`. `SourceControlProvider` interface exposes `get_workflow_metadata()` (branching model description + error recovery flows) and `suggested_primary_branch_name()`. Branch prompt trigger is universal — no operation-specific trigger list needed. | §13.9 |
| 19 | **Authentication and identity model** | No application-level passwords. Each machine has a local profile (username + email) created on first launch, stored in OS app data — the sole source of commit author identity across all projects on that machine. Philotechnia has no project-level user registry, no role system, and no application-layer access control (see #12). A team member's username is stamped into `Record.created_by` at record creation and supplied to the provider as commit author on every commit, giving a durable audit trail through the provider's history. Onboarding a new team member is simply: clone, open, create local profile on first launch (if not already set). No Owner action, no credential exchange. | §13.10 |
| 20 | **Centralized provider offline UX** | Editing always permitted. Commit is disabled with a clear inline explanation when connectivity is unavailable. A persistent connectivity status indicator is shown in the source control panel for providers where `supports_offline_commits()` == false. Must be accommodated in v1 UI groundwork. | §13.11 |
| 21 | **Git hosting compatibility (v1)** | Explicit targets: GitHub, GitLab, Gitea. SSH + HTTPS both required. Credentials delegated to OS credential store via libgit2 callbacks. SSH key path/passphrase configurable in app settings. | §13.12 |
| 22 | **File attachment storage strategy** | `attachment` field stores a URI (URL or local file path) to an externally hosted resource. No binary data in the repo. Known limitation: URIs can become stale. More integrated strategy (Git LFS, etc.) deferred to a future milestone if needed. Available in v1. | §13.13 |
| 23 | **Pull strategy (distributed providers)** | Merge-on-pull is the v1 default. Implemented as `git_remote_fetch` + programmatic merge with manual index handling — the default checkout path that would write conflict markers to disk is never invoked. Each conflict surfaces as three byte blobs (ancestor, local, incoming); JSON-aware field-level merge logic lives in the application core, not in the provider. Rebase-on-pull is deferred. Centralized providers (SVN, Perforce) have no analogous choice. | §6.9, §13.14 |
| 24 | **Commit pipeline failure handling** | Commit-atomicity boundary is the VCS commit, not the filesystem batch. On any write failure the pipeline aborts before calling `stage()` / `commit()` and in-memory state remains dirty; retry completes the batch (atomic rename makes re-writes idempotent). Partial on-disk writes after a crash surface as pending changes via `get_pending_changes()`. `mark_clean()` runs only after `commit()` succeeds. Preflight disk-space check deferred as a UX improvement (candidate for M4). | §7.2, §13.16 |
| 25 | **Format version compatibility** | `format_version` on disk equal to running build: open normally. Lower: upgrade on first commit after opening. Higher: refuse to open with "newer version required" dialog; a read-only fallback is deliberately not offered to avoid partial-format corruption risk. Distinct from schema versioning (see #14). | §7.2 |
| 26 | **Project-open cache** | mtime-based cache of parsed record content stored per-machine and per-project under OS app data (keyed by `project.id`); never inside the project directory, so never committed. Per-entry invalidation on mtime/size mismatch; whole-cache invalidation on `format_version` change, schema fingerprint change (over all files under `schema/`), or cache-format version change. Written only after successful `commit()` — extends CLAUDE.md rule 9 beyond record and schema writes. Orphan caches not opened in 90 days are removed on app startup. | §8.6, §13.17 |
| 27 | **Typed workflow-metadata handshake** | `RecoveryFlow.error_state` and `Step.action_id` are typed `enum class ErrorState` / `enum class ActionId`, not `std::string`. Both enums are application-owned (`src/core/workflow.hpp`); providers import them. `ActionId` resolves through a single dispatch table in `src/core/workflow.cpp` — UI widgets never `switch` on the value. Rule-6 exhaustiveness is enforced by three CI-checked constituents that land with their dependencies: (1) at M1b, a unit test asserts every `ErrorState` value has a `RecoveryFlow` from some shipped provider's `WorkflowMetadata` or is listed in `kUnshippedErrorStates` (v1 uses that escape hatch only for `CentralizedCommitBehind`, removed when SvnProvider ships); (2) at M3–M4, a unit test asserts every `ErrorState` value has a renderer in the recovery UI, with the same exemption applied; (3) at M4, the dispatch function in `src/core/workflow.cpp` is a `switch` over every `ActionId` enumerator with no `default` case, making a missing dispatch entry a compile error. Together these express CLAUDE.md rule 6 as code. | §8.4, §13.18 |
| 28 | **User identity key** | `LocalProfile.username` is the sole identity key across the system — stamped into `Record.created_by` at record creation time and supplied as the provider's commit author name. No UUID-based user identity, no project-level user entity, no second identity space. A future username-rename flow must rewrite `created_by` across every record the user authored in every project that references the old name, so usernames are treated as stable in v1; the rename flow is deferred alongside the "purge deprecated fields" action. | §7.2, §13.19 |
